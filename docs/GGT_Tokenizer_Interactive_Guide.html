GobeloWordTokenizer v1.0.0

Overview

GobeloWordTokenizer is the Phase 1 component of the Gobelo Grammar Toolkit — a language-agnostic word tokeniser for the seven official regional languages of Zambia: ChiBemba, ChiTonga, ChiNyanja, SiLozi, ciLunda, ciLuvale, and ciKaonde.

All language-specific knowledge flows exclusively from external data files. Adding a new language requires zero Python changes. The tokeniser produces AnnotatedSentence objects consumed directly by Phase 2 (GobelloMorphAnalyser).

Phase 1 scope: The tokeniser handles splitting, classification, and lexicon stub lookup. It does not perform morphological analysis — that is Phase 2. Verb slot_parses, morpheme_spans, and feats are all populated downstream.

Quick start

Full production setup

from word_tokenizer import GobeloWordTokenizer

loader     = GobeloGrammarLoader("bem")           # ChiBemba
corpus_cfg = CorpusConfig.load("corpus_config.yaml")

tok = GobeloWordTokenizer(
    loader         = loader,
    corpus_cfg     = corpus_cfg,
    lang_iso       = "bem",
    sent_id_prefix = "bem-GEN-001",
)

sent = tok.tokenize(
    "Bakali balima mu nganda.",
    sent_id = "bem-GEN-001-01",
    source  = "Bible:Mk.1.1",
)

for token in sent.tokens:
    print(token.token_id, token.form, token.upos, token.flags)

Six-stage pipeline

Click a stage to expand its implementation details.

1

Pre-normalisation

NFC normalise · OCR corrections · noise-char stripping

›

Applied to the entire raw string before any splitting occurs:

  1. NFC normalise — unicodedata.normalize("NFC", text). Ensures combining diacritics are in canonical form.
  2. OCR corrections — apply ocr_map substitutions longest-key-first to prevent partial replacements. Source: corpus_cfg.get(lang, "ocr_corrections").
  3. Noise strip — remove any character whose Unicode category is in noise_categories or whose code point is in noise_chars. Source: corpus_cfg.
# corpus_config.yaml example
ocr_corrections:
  "ﬁ": "fi"         # ligature fix
  "\u00AD": ""       # soft hyphen removal
noise_unicode_categories: ["Cf"]  # strip Unicode format chars
2

Special-token pre-scan

Verse references · chapter headings · custom regex patterns

›

Scans the normalised string with each compiled special_pattern. Returns a sorted list of (start, end, TokenType, xpos) tuples. Overlapping spans are resolved: earliest start wins; on tie, longer span wins.

PatternDefault regexTokenTypexpos
Verse ref^\d+:\d+$SPECIALVERSE_REF
Chapter heading^[A-Z][A-Za-z0-9 ]+\s+\d+$SPECIALCHAP_HEAD
Customfrom corpus_configSPECIALCUSTOM

Special spans that exactly cover a whitespace-split chunk are emitted directly in Stage 3, bypassing punct and clitic splitting.

3

Whitespace splitting

Unicode whitespace · original byte offsets preserved

›

A character-walk yields (chunk, start, end) tuples preserving character offsets into the normalised string. Chunks that exactly match a Stage 2 special span are emitted as special tokens directly.

"Bakali balima 2:15." 
→  ("Bakali", 0, 6)
   ("balima", 7, 13)
   ("2:15.", 14, 19)   ← punct splitting will handle the period
4

Punctuation splitting

Sentence-final and inline punct · protected patterns preserved

›

For each non-special chunk, _PunctSplitter:

  1. If chunk matches any protect_pattern (URL, abbreviation like Mk., decimal number) → emit as-is.
  2. Otherwise, scan character-by-character: when a punctuation character from sentence_final_punct ∪ inline_punct is found, flush the buffer as a word-chunk and emit the punct char as its own token.
"nganda."  →  ("nganda", word) + (".", punct)
"2:15."   →  ("2:15.", whole — protected by decimal pattern? No → "2:15" + ".")
"Mk."     →  protected by r"[A-Z][a-z]*\." → emitted whole
Default protect patterns: https?://\S+, \d+\.\d+, [A-Z][a-z]*\., [A-Z]{2,}. Override per-language in corpus_config.
5

Clitic splitting

Proclitics and enclitics · longest-match greedy

›

_CliticSplitter applies to each non-punct word chunk. Clitic lists are sorted longest-first at config construction time for greedy matching.

  1. Proclitics (left): try each in order; detach the first that matches and leaves a non-empty remainder. One proclitic per token in the current implementation.
  2. Enclitics (right): same logic applied from the right. One enclitic per token.
  3. Detached clitics become TokenType.CLITIC tokens; clitic_of is set to the host token's id.
# corpus_config.yaml
extra_enclitics: ["fo", "mo", "ni"]

# Input: "bonamo"
# Enclitic "mo" matches suffix and leaves "bona" (non-empty)
# → WordToken(form="bona") + WordToken(form="mo", type=CLITIC, clitic_of="N")
6

Post-processing

Numerics · reduplification · lexicon probe · code-switch detection

›

Applied to every final word-form chunk in order:

CheckConditionEffect
Numericdigit string, decimal, or Roman numeral I–XXXIXTokenType.NUMBER, upos=NUM, flag NUMERIC
All-punctall chars are Unicode P or S categoryTokenType.PUNCT, upos=PUNCT
Reduplificationform[:n] == form[n:2n] for any n≥2, length≥4is_reduplicated=True, flag REDUPLICATED

Code-switchtone-marking language + no combining diacritics + all-Latin + ≥4 charsTokenType.CODE_SWITCH, flag CODE_SWITCH
MISCalwaysmisc["NFC"] = NFC form

Verb slot model

The GGT models the agglutinative Bantu verb as eleven positional slots. The tokeniser does not fill these slots — that is Phase 2. But it creates the SlotParse stubs and lexicon hints that Phase 2 operates on.

S1
Pre-SM Neg
S2
Subj Marker
S3
TAM / Rel
S4
Obj Marker
S5
Verb Root
S6
Ext A
S7
Ext B
S8
Ext C
S9
Ext D
S10
Final V
S11
Post-FV

Extensions A–D correspond to valency-increasing (Z1), valency-adjusting (Z2), voice (Z3), and aspectual/lexical (Z4) zones. Phase 2 fills these slots with SlotFill objects carrying form, gloss, source rule, and confidence level.

Language-specific notes

ChiBemba (bem)

engine_features.extended_H_spread: true — stored in _cfg.extended_h_spread but acted on by Phase 2 tonal rules, not the tokeniser. NC7 prefix fi-/ifi- and NC8 bi-/ibi- affect lexicon probe noun-class hints. Augment system (i-/a-) present; has_augment=True propagates to the analyser.

SiLozi (loz)

engine_features.extended_H_spread: false — tonal rule TS.2 does not fire. NC16 locative is fa- (not pa-). No augment system; has_augment=False. The ne past auxiliary is handled as a separate whitespace-split token; no special-casing needed.

All languages

Zero Python changes for any new language. Create the GGT YAML, add corpus_config entries, register the language in the loader registry.

Stacking multiple clitics per token

The current implementation handles one proclitic and one enclitic per token. To support stacking, extend _CliticSplitter.split():

def split(self, form: str) -> List[Tuple[str, bool]]:
    pre_clitics, rest = [], form
    # Loop — not just one pass:
    changed = True
    while changed:
        changed = False
        for cli in self._proclitics:
            if rest.startswith(cli) and len(rest) > len(cli):
                pre_clitics.append(cli)
                rest = rest[len(cli):]
                changed = True
                break

Custom code-switch detector (Phase 3 hook)

The current heuristic (Latin + no tone marks) is intentionally conservative. Replace _detect_code_switch() with a language-model based classifier in Phase 3; the TokenType.CODE_SWITCH flag and code_switch_lang field are already in place.
